# المهمة 10: نظام الرفع المتكامل، بيئة Docker، ونظام الدردشة الفورية بـ Socket.IO

تطبيق ويب مبني باستخدام Node.js و Express يعتمد على هيكلية نظيفة وعزل كامل للمنطق البرمجي، يغطي متطلبات إدارة المرفوعات محلياً وسحابياً، الإعداد الحاوي والتشغيل الموحد باستخدام Docker، ونظام محادثة فوري ومزامن.

---

## 1️⃣ إدارة متغيرات البيئة المستعملة (Environment Variables)
لحماية البيانات السرية، تم فصل المتغيرات في ملف `.env` (ويوجد نموذج توضيحي باسم `.env.example` في المستودع). المتغيرات المطلوبة للتشغيل هي:

* `PORT`: المنفذ الافتراضي لتشغيل خادم الويب (مثل `3000`).
* `MONGODB_URL`: رابط الاتصال بقاعدة بيانات MongoDB (يُستخدم المسار المحلي أو اسم الخدمة داخل الحاوية).
* `CLOUDINARY_CLOUD_NAME`: اسم الحساب الخاص بك على منصة التخزين السحابي Cloudinary.
* `CLOUDINARY_API_KEY`: مفتاح الواجهة البرمجية لـ Cloudinary.
* `CLOUDINARY_API_SECRET`: المفتاح السري الخاص بـ Cloudinary.

---

## 2️⃣ طريقة تشغيل المشروع باستخدام Docker
تم إعداد المشروع ليقوم بإقلاع الخدمات وقاعدة البيانات معاً بشكل متزامن ومعزول عبر حاويات دوكر، وذلك من خلال تشغيل الأمر التالي في موجه الأوامر (Terminal) بداخل المجلد الرئيسي:

```bash
docker compose up --build
```

### شرح تفصيلي لآلية عمل الأمر:
* `docker compose up`: يقرأ ملف التنسيق `docker-compose.yml` ويقوم بإنشاء وإقلاع الخدمتين المعرفتين بداخله: خدمة التطبيق البرمجي `api` القائم على بيئة Node.js، وخدمة قاعدة البيانات الرسمية المستقرة `mongo` القائمة على MongoDB، وربطهما تلقائياً بشبكة معزولة ومشاركة ليتم الاتصال بينهما بنجاح دون الحاجة لتثبيت أي منهما محلياً على جهاز المستخدم.
* `--build`: يضمن هذا الخيار إجبار محرك الدوكر على إعادة بناء وصياغة صورة الحاوية (Docker Image) الخاصة بتطبيق النود من الصفر بناءً على كتيب التعليمات بملف الـ `Dockerfile` لتضمين وتحديث أي تعديلات جديدة تمت كتابتها في الأكواد البرمجية قبل تفعيل التشغيل الفعلي.

---

## 3️⃣ خريطة أحداث نظام الدردشة الفورية (Socket.IO Event Map)
تم فصل منطق الأحداث بملف مستقل `src/socket.js` لتعزيز ممارسات الكود النظيف والمستقر، ويعمل النظام وفق خريطة الأحداث التفاعلية التالية:

1. **الأحداث المدمجة (Built-in Lifecycle):**
   * `connection`: يُطلق تلقائياً بمجرد اتصال العميل بسيرفر الـ WebSockets، ويقوم الخادم بطباعة الـ `socket.id` الفريد في الـ Console لتتبع حالة الاتصال الحية.
   * `disconnect`: يُطلق تلقائياً بمجرد إغلاق المتصفح أو قطع العميل لاتصاله، ويقوم السيرفر بإرسال بث لإشعار بقية الأعضاء بخروجه.

2. **الأحداث المخصصة (Custom Chat Events):**
   * `chat:join`: يرسله العميل عند دخوله حاملاً الاسم `{ username }`؛ فيقوم السيرفر بحفظ الاسم بداخل الجلسة البرمجية الحالية وبث إشعار ترحيبي بالنظام لبقية المستخدمين المتصلين.
   * `chat:message`: يرسله العميل عند رغبته بإرسال نص محادثة `{ username, text }`؛ فيقوم الخادم باستقباله وإعادة بثه شمولياً لجميع المتصلين بالدردشة فوراً لتحديث الواجهات في نفس اللحظة.
   * `chat:typing`: يرسله العميل فور بدء كتابته داخل صندوق النص ليقوم السيرفر بنقل وبث حالة مؤشر الكتابة التفاعلية لبقية المتصلين (مثل إظهار عبارة "المستخدم يكتب الآن...").
